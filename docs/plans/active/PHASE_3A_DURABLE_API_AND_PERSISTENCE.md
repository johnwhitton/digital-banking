## Architecture decisions

### One explicit PostgreSQL adapter

- `PostgresOperationRepository` implements the existing application `OperationRepository` using `JdbcClient` and `TransactionTemplate`.
- Acceptance first performs a read optimization for an already committed scoped key, then uses `INSERT ... ON CONFLICT DO NOTHING` as the concurrency authority. The read is never the race-resolution mechanism.
- A proposed operation is created only when no committed binding is visible. Under a true concurrent race, an unused proposed UUID may be generated by the loser, but only the database winner can persist an operation or external-effect identity.
- The scoped key is tenant, participant, resource, operation kind, and SHA-256 key digest. Keys are 1–128 visible US-ASCII characters. Canonicalization version and command digest determine replay versus conflict. The raw key is never logged or stored.
- The idempotency foreign key is deferrable so the winner can claim the scoped identity before inserting the operation in the same transaction. A loser blocks on the uniqueness decision, then reads the committed winner under PostgreSQL `READ COMMITTED` without using an exception that marks its transaction rollback-only.
- The adapter explicitly sets every acceptance transaction to `READ_COMMITTED`, independent of the connection-pool default. Application event timestamps are truncated to PostgreSQL microsecond precision before aggregate creation so original and replay representations remain byte-stable.
- `save` permits exactly one aggregate version advancement, updates `token_operation` with `WHERE version = :expectedVersion`, and appends exactly one transition, attempt, or finality event. Existing history is never updated or deleted through the repository API.
- Reads rebuild `TokenOperation` from the initial requested record and version-ordered transition/attempt/finality events. This reuses domain guards rather than adding a privileged public hydration constructor.

### One deterministic Flyway schema

`V1__create_token_operation_persistence.sql` owns these normalized tables:

1. `token_operation` — immutable acceptance context, exact asset/unit definition, positive `NUMERIC(512,0)` atomic quantity and maximum, lifecycle state/version, safe acceptance evidence, and UTC timestamps.
2. `operation_idempotency` — scoped hashed key, operation kind/resource, canonicalization version/digest, operation identity, and creation time.
3. `operation_transition` plus `operation_transition_evidence` — initial acceptance audit at sequence/version zero and later append-only domain transitions.
4. `operation_attempt` — ordered stable attempt identity, predecessor/retry authorization, evidence, aggregate version, and time.
5. `operation_finality` plus `operation_finality_evidence` — four independent ordered histories, including the initial `NOT_ASSESSED` rows.
6. `operation_outbox` — one pending versioned `TokenOperationAccepted` event, stable event/operation IDs, safe JSONB payload built by PostgreSQL from bound values, and created/available timestamps.

Primary/foreign/unique/check constraints enforce identifiers, kinds, states, four finality types/statuses, positive bounded exact quantities, event ordering, and duplicate-event prevention. Indexes cover participant-scoped reads, idempotency lookup, attempt lineage, finality history, and future pending-outbox claiming. Flyway clean remains disabled.

### Application and status boundary

- Extend `IdempotencyScope` with operation kind so mint and burn resource identities are explicit.
- Extend `OperationRepository` with participant-scoped lookup while retaining internal ID lookup for future lifecycle coordination.
- Add a framework-free `AssetUnitCatalog` port and a small acceptance coordinator that validates request contract version, resolves the server-owned unit definition, parses the canonical quantity, and delegates to the existing `TokenOperationService`.
- Add explicit unsupported-contract, unknown-unit, and not-found application exceptions without HTTP status or database types.
- Keep the default catalog empty. A fixture-only `REFERENCE_ASSET`/`REFERENCE_UNIT` definition is supplied by test configuration and is never represented as a branded token or public-chain asset.

### Security, API, and problem responses

- A control-plane `ParticipantPrincipal` supplies tenant and participant identity from Spring Security. JSON and headers cannot set participant scope.
- Stateless security permits only health/readiness and `/openapi/token-operations-v1.yaml` anonymously. Mint requires `token:mint`, burn `token:burn`, and GET `token:read`; every other request is denied. Form login, HTTP Basic, logout, request caching, sessions, generated credentials, and CSRF are disabled or absent.
- Original acceptance and replay both return 202 with the same stable representation and `Location`; replay is documented and tested, with no duplicate operation/outbox row.
- GET uses participant-scoped storage lookup so unknown and cross-participant identifiers produce the same 404 problem.
- Spring `ProblemDetail` provides RFC 9457 `application/problem+json` bodies with stable `urn:digital-banking:problem:*` types. No request body, key, digest, SQL, stack, class name, credential, or provider detail is returned. Participant responses use an explicit `participant:` evidence-reference allowlist; internal command, authorization, transition, and finality evidence is omitted.
- A single design-first OpenAPI 3.1 document under control-plane resources is the authority and is served at `/openapi/token-operations-v1.yaml`. A direct YAML parser/conformance test checks paths, operations, security, headers, string quantity, four finalities, problems, examples, asynchronous semantics, and absence of public servers. No springdoc, UI, generated source, or second snapshot is introduced.
